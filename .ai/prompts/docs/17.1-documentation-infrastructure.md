# OpenMeta — Phase 17.1: Documentation Infrastructure

You are a Principal Documentation Architect, Technical Writer, and Open Source Framework Maintainer.

You are designing the documentation infrastructure for the OpenMeta framework.

Before making any changes, read: README.md, ARCHITECTURE.md, ROADMAP.md, CONTRIBUTING.md, docs/, docs/adr/.

Project documentation is the single source of truth. Never contradict existing architecture.

---

# Goal

Design the complete documentation infrastructure for OpenMeta. This phase is NOT about writing tutorials. It is about creating the documentation architecture that every future document will follow. Everything should be modular, versionable and maintainable.

# Objectives

Documentation architecture · conventions · folder structure · naming conventions · Markdown standards · frontmatter specification · asset organization · diagram standards · versioning strategy · search strategy · cross-reference strategy · documentation governance.

# Principles

Developer First · Documentation First · Architecture Driven · Version Controlled · Searchable · Consistent · Extensible · Framework Independent.

# Folder Structure

docs/ getting-started, concepts, architecture, packages, api, examples, tutorials, recipes, reference, contributing, release, adr, assets/(images, diagrams, logos, icons, templates), scripts, README.md.

# Responsibilities

Define responsibilities for every documentation folder.

# Documentation Standards

File/folder naming, headings, markdown, tables, lists, code blocks, quotes, callouts, warnings, notes, best practices, examples, images, links.

# Frontmatter

Standard metadata: title, description, package, category, version, status, author, last_updated, tags, difficulty, estimated_time, related_documents. Define the specification only — no tooling.

# Markdown Rules

Heading hierarchy + max depth, line length, code fences + language identifiers, tables, emoji usage, blockquotes, callouts, links (relative/internal/external).

# Code Examples

Compile · complete · explained · reference related docs · no pseudo code · follow coding standards.

# Images

SVG/PNG/screenshots/icons, naming, dimensions, compression, alt text, dark-mode compatibility.

# Diagrams

Mermaid, flowcharts, sequence, ER, package, dependency, architecture. Define standards only — do not generate images.

# Cross References

Related packages/tutorials, API/architecture references, examples, recipes, ADRs.

# Search

Searchable metadata: package, version, tags, difficulty, keywords, categories.

# Documentation Quality

Grammar, terminology, consistency, broken links, missing images/references, outdated/deprecated content.

# Versioning Strategy

Development, Alpha, Beta, Stable, LTS. Older versions remain accessible.

# Templates

Package doc, tutorial, example, recipe, reference page, API overview, architecture page, ADR, release notes, migration guide.

# Governance

Review process, approval, ownership, contribution rules, deprecation policy, archiving policy.

# Deliverables

Documentation Architecture · Folder Structure · Standards · Naming Conventions · Frontmatter Specification · Markdown Style Guide · Diagram Standards · Governance Guide · Template Specifications.

No implementation code. No documentation website. No tutorials. Only infrastructure and standards.
